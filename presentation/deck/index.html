02 / 09
the bet

Porting a small JS UI library to a new runtime should take five commits.

When the library is mostly data-driven and the rendering layer is small, you can share by default and fork only at the JSX + style-value boundary. No rewrites, no fork divergence, no per-spinner duplication. One animation hook runs on both runtimes via a single build-time alias.

react native + expo

The original. <View>, <Text>, StyleSheet.create(), react-native-safe-area-context. 55 spinner components, each owning its own useState / useEffect / FRAMES array.

reactlynx · rspeedy

The port. <view>, <text>, intrinsic elements, dual-thread runtime, CSS strings for style values, className-driven SCSS. One generic Spinner.tsx + 55 thin named exports auto-derived from src/data/.

20 lines of JSX. Four real incompatibilities.

These aren't bridgeable by aliasing; they need different JSX and different style value formats. So we fork. Only here.

react native · src/components/spinners/dots.tsx
import { useEffect, useState } from "react";
import { Text, View } from "react-native";

const FRAMES = ["⠋","⠙","⠹","⠸", ...];
const INTERVAL = 80;

export function DotsSpinner({ size = 24, color = "#fff", style }) {
  const [frame, setFrame] = useState(0);
  useEffect(() => {
    const id = setInterval(
      () => setFrame((i) => (i + 1) % FRAMES.length),
      INTERVAL);
    return () => clearInterval(id);
  }, []);
  return (
    <View style={[{ alignItems: "center" }, style]}>
      <Text style={{ fontSize: size, color, lineHeight: size * 1.3 }}>
        {FRAMES[frame]}
      </Text>
    </View>
  );
}
lynx · src/lynx/Spinner.tsx
import { useSpinnerFrame } from '../hooks/useSpinnerFrame';
import type { SpinnerDefinition } from '../data/types';

export function Spinner({ definition, size = 24, color = '#fff', className }) {
  const frame = useSpinnerFrame(definition.frames, definition.interval);
  return (
    <view className={className} style={{ alignItems: 'center' }}>
      <text style={{
        fontSize: `${size}px`,
        color,
        lineHeight: `${size * 1.3}px`,
        whiteSpace: 'nowrap',
      }}>
        {frame}
      </text>
    </view>
  );
}

// + named exports auto-generated from data/
export function DotsSpinner(props) {
  return <Spinner definition={data.dots} {...props} />;
}

08 / 09
harness engineering · what made the port boring

Eight techniques. Boring by design.

Sharing maximally and forking minimally (slopfork, if you must) only works when the harness around the agent enforces discipline. Here's what we used.

T-01 · architecture-doc-first

Write LYNX_PORT.md before any code.

263 lines: three-layer architecture, decision log, risks-and-mitigations table, and a numbered implementation order. The five US-* commits then executed that order 1:1. The doc is the plan.

T-02 · PRD-driven planning

One PRD file, five user stories.

tasks/prd-lynx-port.md shipped alongside the architecture doc and was deleted only after every US closed (b5db6a7: "delete completed PRD"). Stories were the unit of work.

T-03 · one commit per story

Self-contained. Build-verified. Narrate the artifact.

Every lynx-port: commit message names a runtime-visible output: bundle size, byte-identity check, "runs in LynxExplorer." Compile-only doesn't count.

T-04 · subagent dispatch

Explore agent → lynx-docs MCP.

For Lynx-specific design questions (element model, event handling, style format) we dispatched the Explore subagent and pointed it at the lynx-docs MCP server instead of web search. Logged at subagents/agent-af972c4e0da5ddf28.jsonl.

T-05 · MCP over web search

ListResources → ReadResource → llms.txt entrypoint.

For platform docs (elements, layout, threading), the lynx-docs MCP's llms.txt entrypoint was the canonical index instead of guessing URLs. Faster, no hallucination surface.

T-06 · end-to-end verify, every step

"编译通过 ≠ 能用." Build the example, look at it.

From the user's global CLAUDE.md, repeated in the project CLAUDE.md: "Verify visually in the demo app after every change. pnpm build passing is not sufficient." Caught both post-launch fixes (519ab47, 05b8d51).

T-07 · TaskCreate todo tracking

Tasks are the unit of progress.

Within a story the agent uses TaskCreate / TaskUpdate to split work into atomic pieces (extract data → write hook → write renderer → wire demo → screenshot). Each marked completed when its artifact lands.

T-08 · Ralph autonomous loop

Long-horizon work without a babysitter.

The cleanup commit "remove stale ralph-loop artifacts" confirms the port ran through a Ralph autonomous loop. The agent advanced one US per turn, with the harness blocking stop until the user-defined goal was met — the same hook that shipped this deck.

09 / 09
the playbook

Next time you fork a JS app into a new runtime.

A repeatable recipe, distilled from this port. Boring is the feature.

  1. Architecture doc first. Write the layered map, the decision log, and the numbered implementation order before touching code. The doc is the plan.
  2. Find the fork line. Cluster everything platform-agnostic — data, types, pure hooks — and push it down into one shared tree. Fork only at the JSX + style-value boundary.
  3. One build-time alias replaces one platform shim. 'react' → '@lynx-js/react' unlocks the shared hook. Prefer aliasing over abstracting.
  4. One commit per user story. Always end with a runnable artifact. Bundle size, screenshot, byte-identity check. Never "tests pass."
  5. Subagents for platform docs. Dispatch Explore + the platform's MCP (here: lynx-docs) for element reference, instead of web-searching.
  6. Verify visually, every step. Bracket every "feature complete" claim with a screenshot or recording of the running demo. Compile is not done.
  7. Run the agent in a Ralph loop with a goal hook. One US per turn, harness blocks stop until the goal is met. Walk away, come back to a finished port.
